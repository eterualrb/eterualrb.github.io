直播总结一 FLV文件分析

2017.06.13 王白告

一、概述

FLV(Flash Video)是Adobe公司设计开发的一种流行的流媒体格式，由于其视频文件体积轻巧、封装简单等特点，使其很适合在互联网上进行应用。此外，FLV可以使用Flash Player进行播放，而Flash Player插件已经安装在全世界绝大部分浏览器上，这使得通过网页播放FLV视频十分容易。目前主流的视频网站无一例外地使用了FLV格式

二、FLV文件结构

FLV封装格式的文件后缀通常为“.flv”，总体上看，FLV包括文件头（File Header）和文件体（File Body）两部分，其中文件体由一系列的Tag组成，每个Tag前面还包含了Previous Tag Size字段，表示前面一个Tag的大小。Tag的类型可以是视频、音频和Script，每个Tag只能包含以上三种类型的数据中的一种

1. FLV File Header

从上图我们可以看出FLV Header的结构是固定的9个字节，这里就不再详细记录了，再来个官方文档的介绍来加深记忆吧

2. FLV File Body

接下来我们来分析FLV File Body的结构，前面说过FLV File Body是由一个个的Tag组成，所以我们着重来分析Tag的结构就可以了，同样先来个官方文档的介绍吧

总结如下表所示

Field type&&size Comment
TAG类型 UI8 8 : audio tag
9 : video tag
18 : script data(描述信息)
all others : reserved
数据大小 UI24 数据区的大小，不包括包头。包头总大小是11(1+3+3+1+3)个字节
时戳 UI24 当前帧时戳，单位是毫秒。相对于FLV文件的第一个TAG时戳。第一个tag的时戳总是0。不是时戳增量，rtmp中是时戳增量
时戳扩展字段 UI8 如果时戳大于0xFFFFFF，将会使用这个字节。这个字节是时戳的高8位，上面的三个字节是低24位
流ID U24 总是0
数据区 UI8[n] 对应类型的具体数据内容

从上面可知每个tag不包括数据区总是固定的11个字节，接下来的数据区对应具体的audio data，video data和script data object。我们再来分析数据区的具体内容

2.1 Script Data Tag(控制帧)

该类型Tag又通常被称为Metadata Tag，会放一些关于FLV视频和音频的元数据信息如：duration、width、height等。通常该类型Tag会跟在File Header后面作为第一个Tag出现，而且只有一个。结构如下图所示

第一个AMF包：

第1个字节表示AMF包类型，一般总是0x02，表示字符串。第2-3个字节为UI16类型值，标识字符串的长度，一般总是0x000A（“onMetaData”长度）。后面字节为具体的字符串，一般总为“onMetaData”（6F,6E,4D,65,74,61,44,61,74,61）

第二个AMF包：

第1个字节表示AMF包类型，一般总是0x08，表示数组。第2-5个字节为UI32类型值，表示数组元素的个数。后面即为各数组元素的封装，数组元素为元素名称和值组成的对。常见的数组元素在FLV的官方规范文档中也有描述

这部分的具体细节可以参考另一篇blog直播总结二 AMF编码进行学习

2.2 Audio Tag(音频Tag)

我们来看下官方规范文档中是怎样描述它的信息的

总结如下表所示

Field type&&size Comment
音频格式 UB4 常见的格式有
0 = Linear PCM, platform endian
1 = ADPCM
2 = MP3
3 = Linear PCM, little endian
10 = AAC
采样率 UB2 0 = 5.5-kHz
1 = 11-kHz
2 = 22-kHz
3 = 44-kHz (AAC总是3)
采样大小 UB1 0 = snd8Bit
1 = snd16Bit
声道 UB1 0=单声道
1=立体声,双声道 (AAC总是1)
声音数据 UI8[N] 如果是PCM线性数据，存储的时候每个16bit小端存储，有符号
如果音频格式是AAC，则存储的数据是AAC AUDIO DATA，否则为线性数组

由于后面我们的音频编码格式为AAC(Advanced Audio Coding，高级音频编码)，所以我们还要了解下AAC AUDIO DATA的结构，同样先看下官方文档中的描述

总结如下表所示

Field type&&size Comment
AAC Packet Type UI8 0 = AAC序列头
1= AAC原始数据
AAC Data UI8[N] 数据类型由AAC Packet Type决定
AAC Packet Type = 0 ：Audio Specific Config
AAC Packet Type = 1 ：AAC原始数据

那么Audio Specific Config具体的结构又是什么呢？我们从ISO-14496-3文档中找到了它的定义

由于内容过长，从网上找到其他的定义来看下它的结构

它的结构比较复杂，简化结构如下

AAC sequence header

Audio Specific Config的作用是啥？
答：AAC格式里有个复杂的AudioSpecificConfig, 在FLV格式里称为AAC sequence header。在正式播放ADTS(Audio Data Transport Stream) AAC数据包之前，需要用AudioSpecificConfig生成一个audio_tag交给播放器，解码器会将AudioSpecificConfig中的数据(主要是profile(LC,Main,HE)，sample_rate，channel)生成一个ADTS头(7字节)，后续的Raw AAC Frame Data到来时，就在它的前面插入这7个字节的header信息，这样解码器才能正确的播放AAC音频数据

Raw AAC Frame Data就是具体的AAC数据了，至此AAC格式的Audio TAG就分析完了

2.3 Video Tag(视频Tag)

我们还是来看下官方规范文档中是怎样描述它的信息的

总结如下表所示(这里我们只关心AVC编码格式)

Field type&&size Comment
帧类型 UB4 1 : keyframe (for AVC, a seekable frame)——h264的IDR，关键帧，可重入帧
2 : inter frame (for AVC, a non- seekable frame)——h264的普通帧
编码ID UB4 7: AVC
视频数据 UI[N] 如果是avc，则是 AVCVIDEOPACKET

同样由于后面要使用AVC进行视频编码，所以需要关注下AVCVIDEOPACKET的结构，下面是官方文档的描述

总结如下表所示

Field type&&size Comment
AVC Packet Type UI8 0 ：AVC序列头
1 ：AVC NALU单元
2 ：AVC序列结束
CTS SI24 AVC Packet Type = 0，则为0
AVC Packet Type = 1，则为具体的偏移量
数据 UI8[n] AVC Packet Type = 0，则为解码器配置，sps，pps等相关信息
AVC Packet Type = 1，则为nalu单元，可以是多个
